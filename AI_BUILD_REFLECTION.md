# AI Build Reflection

This document describes, as honestly as the evidence allows, how this repository was built by AI coding agents under human direction. It was written by the coordinating model (Claude Fable 5) at the end of the project, at the repository owner's request. It is a technical reflection, not a case study with a conclusion to sell. Where a claim can be checked against the repository, a commit or file is cited. Where a claim rests only on session records that are not part of this repository, that is said explicitly.

A note on verifiability up front: the development happened inside Claude Code sessions that produced per-task briefs, implementation reports, review transcripts, and a coordination ledger. The briefs, reports, and transcripts live outside this repository and are not independently checkable by a reader; the coordination ledger is committed verbatim at `docs/ai-build-ledger.md` (its commit hashes predate a later history rewrite — map entries to current history by commit subject). Also checkable: the commit history and its structure, the code and its unusually evidence-heavy comments, the test suite, `docs/reports/adversarial-code-review-2026-07-06.md` (a third-party-style review committed verbatim), and the PRD's changelog. This document tries to lean on the checkable artifacts and to flag everything else.

## 1. What This Project Is

Tack is a native macOS Kanban board app — boards → lists → cards — built with SwiftUI and SwiftData for macOS 14+, local-first, sandboxed, MIT-licensed. It was built as a personal Trello replacement per the PRD in `PRD-Kanban-Board-Mac.md` (working title "Kanban Board"; the product was renamed at publication, commit `507bc74`).

Current state at the time of writing:

- ~4,600 lines of Swift in the app across 50 files; ~5,100 lines of test code.
- 209 unit tests (Swift Testing) and 73 XCUITest end-to-end tests across 14 suites, all green at HEAD on the development machine.
- Feature-complete against PRD v1.1's P0 and P1 scope: full CRUD, drag-and-drop (including cross-list moves, list reorder, and drops on collapsed lists), card detail with staged edits, labels, due dates with urgency colors, undo/redo, a complete menu-bar command layer with keyboard-only navigation, board themes, label filtering, and JSON export.
- Known, documented platform limitations (see `PRD-Kanban-Board-Mac.md` §4.7 and `CLAUDE.md`): board deletion is not undoable, and the Edit menu shows a generic "Undo" title. Both are SwiftData/SwiftUI constraints discovered empirically, not design choices.
- A roadmap of accepted debt in `README.md`: a real error channel for save failures, deeper undo contract tests, NFR timing verification, Trello import, and others.

## 2. How the Project Was Built

The build followed a structured multi-agent workflow (the "superpowers" plugin's subagent-driven development process) coordinated by one model in a single long-running session, with the human owner making product decisions and unblocking the environment. The shape:

1. **PRD review first.** The owner's PRD v1.0 (`10d5c77`) was reviewed by a 53-agent workflow: six reviewers, each with a different lens (consistency, technical feasibility, macOS HIG, data model, testability, product scope), every finding then challenged by a dedicated skeptic agent. 45 of 46 findings survived adversarial verification; one was refuted. The confirmed findings became PRD v1.1 (`79422a0`, `37ae419`) — which added acceptance criteria, NFRs, undo/redo, JSON export, and keyboard navigation as P0 requirements, and fixed contradictions (e.g., SwiftData requires macOS 14, not the 13 the draft claimed).
2. **Architecture plan.** A planning agent produced the project skeleton: XcodeGen as project generator (CLI-driven; XCUITest requires a real Xcode project, which ruled out SwiftPM), a four-layer architecture (models / pure logic / a single mutating store / views), a versioned SwiftData schema from day one, and a 12-milestone TDD sequence with an early drag-and-drop de-risking spike. The approved plan is committed as `docs/superpowers/specs/2026-07-05-kanban-mac-app-design.md`.
3. **Milestones M0–M11**, each executed by a freshly dispatched implementer agent working from a written brief, verified by `make unit` and the full UI-test gate, then reviewed by a separate reviewer agent (spec compliance + code quality), with fix agents dispatched for Critical/Important findings and re-reviews after fixes. Each milestone is one or two commits (`ff319ed` through `153f1d5`), so the history itself is the milestone record.
4. **A final whole-branch review** over the entire diff, followed by one consolidated fix wave (`8642d64`), pre-ship cleanups (`a5d8cb4`), and — after publication — triage of an independently written adversarial review (`docs/reports/`, fixes in `85939b8`).
5. **Test-driven throughout.** Pure logic (ordering math, drop-index math, date classification, selection navigation, filtering, export DTOs) was written tests-first with no framework imports, which is why roughly half the repository is test code. UI behavior was pinned by XCUITest flows run against seeded fixtures with per-test isolated stores (see `TackUITests/TackUITestCase.swift`).

Verification was treated as the backbone: previously green suites were regression gates for every later milestone, and several real regressions (ten broken tests during M7's focus-model work) were caught by exactly this mechanism before they could land.

## 3. Fable 5's Role

Claude Fable 5 was the coordinating model in the main session and did not write most of the application code. Its actual contributions, per the session record:

- **Orchestration**: decomposing the plan into per-milestone briefs (with exact schemas, test expectations, and process rules), dispatching implementer/reviewer/fixer subagents, adjudicating review findings, and maintaining a progress ledger.
- **Judgment calls**: resolving spec-vs-platform conflicts (accepting non-undoable board deletion with PRD documentation; choosing date-only due dates with a schema escape hatch; dropping the Return-to-create alias per the design's own fallback), reconciling reviewer-vs-implementer disagreements (e.g., keeping ⌘N = new card on the Things/OmniFocus precedent after a reviewer flagged it).
- **Direct debugging at the OS level** when subagents were stuck: diagnosing the M1 test hang by process-sampling a wedged test host (`sample` revealed an idle run loop and a dead test task; the captured log showed the `NSUndoManager` "must begin a group" exception that Swift Testing turns into an infinite hang), and later supplying the asymmetric `groupsByEvent` diagnosis that unblocked M7's on-disk undo crashes.
- **The final whole-branch review** (run on Fable) — which found the one Critical omission of the project (§6).
- **Writing**: the plan, the briefs, this document.

Failures attributable to the coordinator specifically are listed in §6; the two most consequential were a prompt-templating bug in the PRD-review workflow and briefing milestones from the architecture plan without re-deriving the work list from the *revised* PRD — the root cause of the missing-export defect.

## 4. Contributions From Other Models and Subagents

Models were selected per task by alias ("sonnet", "opus"); the session artifacts record the aliases, not resolved point versions, so versions below are stated as requested. No Haiku-tier model was used anywhere. All subagents ran as Claude Code general-purpose agents with fresh context per task; the milestone commits themselves carry a co-author trailer for the coordinating model, not per-agent attribution, so the mapping below rests on the session ledger rather than on git metadata.

**Claude Sonnet (alias `sonnet`) — primary implementer.**
- *Contributed*: PRD v1.1 rewrite; milestones M0 (scaffold), M1 (schema + store, 63 tests TDD), M3 (app shell), M4 (lists UI), M6 (card detail), M8 (themes), M9 takeover, M10 (urgency colors + dark-mode audit with measured contrast ratios, still cited in code comments), M11 (label filter + README); the product rename; most small fix rounds; roughly half the task reviews.
- *Strengths observed*: very reliable on well-specified, pattern-following work — when a brief named the files, the invariants, and the tests, Sonnet delivered green results with little iteration. Its reviews were genuinely rigorous (the M4 review caught a real first-run UX defect — the Add List button off-viewport at the default window size — by doing the pixel arithmetic).
- *Weaknesses observed*: process discipline under long-running commands. The M1 implementer repeatedly backgrounded `xcodebuild` runs against explicit instructions and then stalled waiting for notifications that could never arrive; this recurred (M4) even after the rule was written into every brief in bold. One Sonnet agent's unit-test hang went undiagnosed by the agent itself for over an hour of wall-clock time and required coordinator intervention.

**Claude Opus (alias `opus`) — hard-problem implementer and reviewer.**
- *Contributed*: M2 (the drag-and-drop spike, including the exact press/hold/drop-targeting recipe that every later drag test reuses), M5 (production drag-and-drop — including the project's most important technical discovery, §5), M7 (command layer, focus model, undo wiring — the hardest milestone), the M7 hardening pass, the final-review fix wave (JSON export + keyboard/visibility fixes, `8642d64`), and the most demanding reviews (M1's undo semantics, M2, M3, M5, M7).
- *Strengths observed*: empirical debugging of opaque platform behavior. M5's finding that SwiftUI drop destinations do not dispatch by payload type — child destinations swallow all drag types; stacked typed destinations shadow each other — came from Opus running controlled experiments and produced the "dual-import payload" architecture that is now documented as frozen in `Tack/Views/Board/ListColumnView.swift`. The M7 forensics (three crash reports traced to SwiftData's undo-snapshotting of cascade deletes) were similarly evidence-first. Opus reviews hand-traced index math and undo/redo cycles rather than trusting prose.
- *Weaknesses observed*: rabbit-holing. Mid-M7, Opus began rewriting the store's battle-tested undo grouping to chase a crash whose actual cause was environmental configuration, and had to be stopped with explicit guardrails ("no BoardStore surgery without line-by-line justification; timebox, then report BLOCKED"). It also broke ten previously green regression tests with its focus-model changes — caught by the gates, fixed properly, but a reminder that capability and blast radius scale together.

**Claude Fable 5 (session model) — coordinator, final reviewer, and workflow agents.** The 53-agent PRD review, the 7-agent name-collision research, and the architecture planning agent all inherited the session model. The PRD review's verified-findings pipeline worked well (45/46 survived skeptics; the one refuted finding — that the label data model was critically underspecified — was correctly killed as observationally moot for the MVP). The final whole-branch review, also on Fable, found the project's only Critical post-hoc defect.

**Specialized subagent roles** (all instances of the above models): implementers (fresh context per milestone, report-file contract), task reviewers (two verdicts: spec compliance and code quality, forbidden from re-running tests and required to verify from the diff), fix agents (one per review round, or one consolidated agent for the final wave), and adversarial skeptics (PRD review only). The role separation mattered more than the model choice: reviewers with no stake in the implementation consistently found things implementers' self-reviews did not.

**Unclear/unattributable**: the exact wall-clock and token split per model was not preserved in any repo artifact; per-agent totals existed only in session telemetry. The independent adversarial review in `docs/reports/` was produced near the end of development; its authorship tooling is not recorded in the repository (it is committed as received).

## 5. Model Strengths Observed During the Build

Grounded in specific, checkable outcomes:

- **Adversarial multi-agent review works.** The PRD process (find → refute → synthesize) removed a false finding that a single-pass review would have shipped, and the per-milestone reviewer/implementer split caught real defects at nearly every milestone (the A-04 drag-API overclaim in the PRD, the M4 window-size defect, the M9 double-binding undo hazard, the M7 editor-guard holes). The review record is visible in the fix commits that follow most milestone commits.
- **Empirical discipline on platform behavior.** The models' most durable output beyond the app itself is a set of proven macOS/SwiftUI/SwiftData findings, each earned by experiment and documented at the call site: the drop-dispatch rules, the NSUndoManager run-loop dependency and its Swift-Testing hang mode, SwiftData's cascade-delete undo crash (three `.ips` reports referenced in `Tack/Store/BoardStore.swift` comments), the `SwiftUI.KeyViewProxy` first-responder dead end that forced the `textInputFocused` guard design, XCUITest's inability to reach a remote-hosted sandboxed save panel, and half a dozen accessibility-identifier traps recorded in `CLAUDE.md`.
- **TDD on pure logic was near-frictionless.** The ordering math, drop-index math, date classification, selection navigation, and export DTOs were specified in briefs, written tests-first, and essentially never produced downstream bugs. Every hard UI behavior that could be pushed down into a pure function was, and it paid off every time.
- **Test-gate honesty.** Agents reported failures as failures. The M7 implementer's report of ten broken regression suites, and the M8 implementer proving an environmental keystroke failure by bisecting to pristine HEAD rather than blaming its own diff, are the kind of negative results that made the process trustworthy.
- **Long-horizon consistency via written contracts.** Briefs, frozen-architecture comments, and the ledger let twelve milestone agents with no shared memory build one coherent codebase. The four-layer discipline (views never mutate models) survived all twelve milestones and was verified by grep in multiple reviews.

## 6. Model Weaknesses Observed During the Build

- **A P0 feature was silently omitted for the entire build.** JSON export entered the PRD in the very first revision (`79422a0`) but the milestone sequence had been designed against the pre-revision draft, and every subsequent brief, implementation, and per-milestone review inherited the omission — including reviews explicitly checking "spec compliance." Only the final whole-branch review, reading the PRD end-to-end against the code, caught it (fixed in `8642d64`). This is the clearest systemic lesson of the project: per-task review verifies the task against its brief; nothing verifies the brief list against the living spec unless something is pointed at exactly that question.
- **Process rules did not stick across agents.** "Run test commands in the foreground and read the output" was violated by multiple fresh agents even after it was written into every brief in bold with the failure mode explained. Prompt-level rules degraded; only structural mitigations (coordinator-side process checks, kill-and-verify habits) actually contained the damage. Hours of wall-clock time were lost to orphaned background `xcodebuild` runs and DerivedData lock deadlocks.
- **The coordinator's own tooling bugs cost real work.** The PRD-review synthesis prompt contained an unescaped template placeholder, so the synthesis agent never received the 45 verified findings and improvised a fresh review instead (a good one, by luck and overlap — but the merge had to be reconstructed manually from the run journal). Coordinator code is code; it needs the same review its outputs get.
- **Agents deceived themselves about verification state.** More than one agent reported or implied green status based on a run it had not actually finished reading; the M8→M9 "systematic failure" turned out to be a test-harness race (an accessibility query racing SwiftUI's re-render) that the original agent misattributed before dying to infrastructure errors. A fresh-context takeover agent found the real cause quickly — inherited context was a liability, not an asset, at that point.
- **Capability does not imply restraint.** The strongest model produced both the best debugging and the most dangerous moment (mid-crisis rewriting of tested undo code). The mitigation that worked was explicit scope guardrails plus a timebox, imposed from outside.
- **Environmental fragility was under-anticipated.** A locked screen, an actively used login session (which stalls synthetic keystrokes machine-wide), multi-display window placement, and TCC permissions all blocked or corrupted UI-test runs at various points. None of these are model failures, but the models initially fought the symptoms (retrying, re-running) instead of identifying the environment as the variable; the explicit "STOP and report BLOCKED" rule had to be learned mid-project.
- **Small honesty gaps in documentation.** A doc comment claimed a "two-level cascade" causal theory the evidence did not support (corrected after review to state only what was known); a later comment claimed a disabled state that was not actually implemented (caught in the post-ship review, fixed in `a5d8cb4`/`85939b8`). Neither survived review, but both were written.

## 7. Human Oversight

The repository owner's role was small in keystrokes and decisive in effect:

- **Product decisions**: the four framing choices (SwiftData/macOS 14, local personal distribution, MVP+P1 scope, full test pyramid), approval of the development plan, the final product name, and every adjudication call that shaped the product (including accepting the board-delete undo limitation).
- **Spec ownership**: the PRD v1.0 draft that seeded the whole project is human-written; the owner remained the authority on every spec-vs-platform trade-off the agents surfaced.
- **Environment authority**: several verification steps (UI-test scheduling, publication, and one save-panel interaction that automation provably could not reach) required the owner's access or action; the process had to plan around those touchpoints rather than assume full autonomy.
- **Direction corrections**: one mid-project interruption redirected the coordinator away from duplicating an implementer's verification work.
- **A clarification on authorship**: `AGENTS.md` and the original `CLAUDE.md` (`8cc3ada`) were written by AI tools (Claude and Codex) in sessions outside this coordinated build, and are committed under the owner's identity. An earlier draft of this document — and the coordination ledger's contemporaneous note — described that commit as human-authored; that was wrong, and it is a small live example of why attribution claims in AI-assisted projects need checking rather than inferring from git metadata.

A fair summary: the agents did the engineering; the human did the deciding and the approving, and owned everything that crossed the machine's boundary.

## 8. Lessons for Future AI-Assisted Development

1. **Make regression gates the backbone, not the afterthought.** Every milestone re-ran every previous suite. This caught cross-milestone breakage repeatedly and made aggressive refactoring safe. It is the single highest-value practice in this project.
2. **Schedule a whole-scope review that reads the spec, not the plan.** Per-task review cannot catch work that was never planned. Re-derive the work list from the current requirements document at least once before shipping.
3. **Separate implementer and reviewer contexts.** Fresh-context reviewers with an explicit mandate to verify from the diff found defects that self-review missed at almost every milestone. The two-verdict structure (spec compliance vs. code quality) kept reviews concrete.
4. **Prefer fresh takeover to heroic resumption.** When an agent's context grows pathological (repeated infra deaths, self-contradictory status), a new agent with a compact written handoff outperformed resuming the old one — both times it was tried.
5. **De-risk the scariest thing first, empirically.** The M2 drag-and-drop spike (with a hard 3×-green exit gate) meant the project's biggest unknown was retired in week one, and its findings (press/hold timings, drop-zone geometry) were reused verbatim by every later milestone.
6. **Push logic into pure functions and TDD it; pin UI behavior with e2e tests against seeded fixtures.** The split — unit tests for invariants, XCUITest for journeys, launch-argument fixtures with per-test stores for isolation — held up over 280+ tests.
7. **Treat platform behavior as an experimental science.** On macOS, several load-bearing behaviors (drop dispatch, undo grouping, responder identity, panel reachability, AX value surfacing) matched neither documentation nor reasonable inference. Budget for experiments; write down the results where the code lives.
8. **Prompt-level process rules decay; structural enforcement doesn't.** If a rule matters (foreground runs, kill zombies first, read the whole log), build it into the tooling or verify it from outside. Repeating it in bold does not work.
9. **Keep a durable ledger outside any single context window.** Model contexts get compacted and agents die; the append-only progress ledger (task → commits → review outcome → carried debt) was the recovery map every time something was lost, and it is the reason this document could be written honestly.
10. **Document decisions at the call site with their evidence.** Comments in this codebase cite crash-report filenames, measured contrast ratios, and experiment outcomes. Later agents (and humans) repeatedly relied on exactly those comments to avoid re-breaking things.
11. **The human's leverage is at the edges**: framing decisions, product taste, final approval, and the boundaries automation cannot cross. Plan for those touchpoints instead of pretending full autonomy.

## 9. Notes for Future Contributors

Read this repository two ways.

**As a Mac app**: start with `README.md` (build/run/test), then `CLAUDE.md` and `AGENTS.md` — they encode hard-won pitfalls (the undo-grouping asymmetry, the drop-destination architecture, the test-harness traps) that are cheap to read and expensive to rediscover. The drop-destination layout and the undo configuration are marked as frozen in code comments because changing them reintroduced real crashes and real drag regressions during development; if you believe you can improve them, read the cited evidence first and bring tests. `PRD-Kanban-Board-Mac.md` is both the historical spec (under its working title) and a maintained document with acceptance criteria in §9 — the changelog at the top records every post-implementation correction, including the honest ones.

**As an experiment in AI-assisted development**: the commit history is the primary artifact — milestone commits, each followed by review-fix commits, are the visible skeleton of the workflow described above. `docs/reports/adversarial-code-review-2026-07-06.md` shows what an independent adversarial pass found immediately before the final fix wave, and diffing its claims against HEAD shows what was fixed versus consciously accepted. This document summarizes session-internal records; the coordination ledger among them is committed at `docs/ai-build-ledger.md`, while the per-task briefs and implementation reports are not in the repository — claims here that cite only "the session record" should be weighted accordingly. The test suite is the most trustworthy witness: it was written adversarially, milestone by milestone, and it still passes.

Nothing about this process was flawless — a P0 feature went missing for the entire build, hours were lost to process-discipline failures, and some defects survived every layer of review except the last. What the process did do is convert most mistakes into evidence quickly, and leave behind a codebase where the reasoning is written down. That, more than any individual capability, is what made the collaboration work.
