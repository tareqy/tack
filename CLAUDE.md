# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Native macOS Tack app — a kanban board, Trello replacement: SwiftUI + SwiftData, macOS 14+, single-user, local-only, App Sandbox on. The spec is `PRD-Kanban-Board-Mac.md` (feature IDs like C-08, U-01 referenced in commits/comments come from it); the design doc is `docs/superpowers/specs/2026-07-05-kanban-mac-app-design.md`.

## Commands

`Tack.xcodeproj` is **gitignored and generated** — after adding/removing/renaming source files or editing `project.yml`, run:

```sh
make gen          # xcodegen generate
```

```sh
make build        # build the app
make unit         # unit tests (TackTests, Swift Testing)
make ui           # UI tests (TackUITests, XCUITest; serial, result bundle in .build/results/)
make test         # unit + ui
```

Single test (note `DEVELOPER_DIR` — `xcode-select` on this machine points at Command Line Tools only, so every bare `xcodebuild` needs it; the Makefile exports it for you):

```sh
DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer \
  xcodebuild -project Tack.xcodeproj -scheme Tack \
  -destination "platform=macOS,arch=$(uname -m)" -derivedDataPath .build/DerivedData \
  -only-testing:TackTests/BoardStoreCardTests test          # a suite
# -only-testing:TackTests/BoardStoreCardTests/addCardAppends  # one test
# -only-testing:TackUITests/CardCRUDUITests -parallel-testing-enabled NO  # a UI suite
```

Any ad-hoc `-only-testing:TackUITests/...` invocation MUST carry `-parallel-testing-enabled NO` (the Makefile's `ui` target has it). Without it, parallel test clones fight over app activation, which scrambles launch focus and menu enablement — producing **false failures** in keyboard/menu-gated tests that pass serially (burned an afternoon of bisecting on 2026-07-08).

Manual hand-test against a scratch store (never touches real data):

```sh
open .build/DerivedData/Build/Products/Debug/Tack.app --args --uitest --fixture standard --store-name scratch --reset
```

### xcodebuild process rules

- Run `xcodebuild` in the **foreground** and read complete output. Background runs orphan themselves; concurrent runs deadlock on the shared DerivedData lock.
- Before any run: `pkill -f xcodebuild; pkill -f Tack.app`.
- A unit-test run past ~6 minutes is a **hang, not a slow run** (classically an NSUndoManager exception killing the Swift Testing runner — see Pitfalls). Kill it and read the log tail for a FAULT line.
- UI runs are legitimately slow: `make ui`/`make test` ≈ 20–25 min (74 tests, serial). `tee` xcodebuild output to a log under `.build/` and read it to completion — long foreground commands can get auto-backgrounded, and an unread gate is not a gate.
- UI-test failures on this machine can be **environmental**: launch window-waits stall (log mtime frozen >10 min → kill both processes, retry the whole run once; the same test stalling twice is real), and real desktop notifications break tests (an SMS AutoFill popup overlaid the sidebar mid-run and typed the code into the filter field). Check the failure's xcresult screen recording before debugging, and prefer one clean single-test re-run. Piping through `tee` also masks xcodebuild's exit code — judge a run only by its log.
- A second environmental mode (2026-07-08): the host drifts into a state where **every menu-enablement/keyboard-driven test fails deterministically** (bare arrows, ⌘F, `menuItem(...).isEnabled` asserts) while **all mouse-driven suites pass** — the xcresult recording shows text-gated View-menu items grayed out right after launch, i.e. the sidebar filter field holds text focus at XCUITest launch. Suspected trigger: heavy System Events keystroke scripting + repeatedly `pkill`'d Tack instances in one login session; clearing saved-state and modifier key-ups did NOT recover it — a fresh login session is the recovery path. **Triage rule: before debugging red keyboard/menu tests, run one of them against committed (known-green) code as a control** — if the control fails, it's the environment, stop debugging app code.
- A third environmental mode (M-F Task 1a): **wedged `testmanagerd`** (hours of accumulated CPU) stalls every test runner at "hung before establishing connection" — `pkill testmanagerd` fixes it instantly. Separately, the UI-automation grant can drop mid-session: the runner times out "enabling automation mode" (control-run committed code to confirm before assuming an app regression); `DevToolsSecurity -status` may read disabled even while automation actually works, so probe with one real test rather than trusting it — recovery needs the user to re-grant.
- Reading an xcresult's evidence: `xcrun xcresulttool export attachments --path <bundle>.xcresult --output-path <dir>` (needs `DEVELOPER_DIR`), then pull frames from the exported `.mp4` with `ffmpeg -ss <sec> -i <mp4> -frames:v 1 out.png`.

## Architecture

Three targets (defined in `project.yml`): `Tack` (app), `TackTests` (Swift Testing, unit), `TackUITests` (XCUITest — also compiles `Tack/Support/AccessibilityID.swift` so test code shares the app's identifier constants).

### Layers

- `Tack/Models/` — SwiftData `@Model` entities (Board → BoardList → Card → ChecklistItem, plus CardLabel and M-F's Area — sidebar groups; Board.area is an optional to-one released by Area's .nullify delete rule) under a **versioned schema** (`TackSchema.swift`: `TackSchemaV1` + migration plan). Schema changes need a new version + migration stage.
- `Tack/Store/BoardStore.swift` — the **only mutation surface** for the model graph. Every method wraps its writes + save in exactly one explicit undo group (`withUndoGroup`), so each store call is one user-facing ⌘Z step. Pure ordering/selection logic lives beside it (`Reordering` — including `movedWithinSubset` — `SelectionNavigation`, `SelectionRestore`, `DropMath` in DragDrop) as free functions unit-tested without a container. Views never write to `ModelContext` directly.
- `Tack/Store/ModelContainerFactory.swift` — three containers: `production()` (on-disk), `inMemory()` (unit tests), `uiTest(storeName:reset:)` (on-disk per-test store under the sandbox's `Application Support/UITest/`).
- `Tack/Views/` — `RootView` (NavigationSplitView: `Sidebar/` + `Board/`), `CardDetail/`, shared `Components/`. `Views/Spike/` is a minimal board kept alive as the drag-and-drop e2e regression path. `Components/DesignSystem.swift` holds the surface tokens (`Color.cardSurface`/`columnSurface`/`insetSurface`/`surfaceHairline`), `HoverHighlightButtonStyle`, and the shared `LabelChipLabel` — new UI must use these, not ad-hoc `secondary.opacity(...)` washes (those inverted the card/column figure-ground in light mode; removed in the M12 UI-polish pass).
- `Tack/Commands/` — menu-bar commands (`AppCommands`), attached at the WindowGroup **scene** level (commands registered inside the split view never register). `FocusedValues.textInputFocused` gates editing commands (see Pitfalls).
- `Tack/DragDrop/` — `Transferable` payloads (`CardTransfer`, `ListTransfer`) + `DropMath`. **Architecture invariant:** SwiftUI `.dropDestination` does not dispatch by payload type — a destination swallows every drag landing on it, and stacked different-typed destinations shadow each other. Hence: column container accepts ListTransfer only, card rows accept CardTransfer, and the list footer is ONE dual-import destination (`ColumnDropPayload`) that routes both. Do not refactor back to "one typed destination per payload". (Sidebar board reorder deliberately uses native `List` `.onMove` instead of this machinery — see Pitfalls.)

### Launch paths (`TackApp.init` branches on `AppLaunchConfig`)

- **Production:** on-disk container; failure to open shows `DatabaseErrorView` instead of crashing.
- **`--uitest`:** per-test on-disk store (`--store-name`), seeded by `FixtureSeeder` (`--fixture standard|spike|...`), optionally wiped (`--reset`), appearance forced via `--appearance light|dark`. Animations are stripped for determinism. `--export-to <file>` makes the launch write its own JSON export self-check (E-01); `--import-from <file>` (with `--import-mode add|replace|ask`, default `add`) makes the launch import that file on appear — `add`/`replace` import directly for deterministic content tests, `ask` presents the real mode dialog for a test to drive (E-02).
- **`--uitest --fixture spike`:** boots `SpikeBoardView` directly (drag regression path).

### UI-test harness

Subclass `TackUITestCase`. `launch(fixture:)` gives each test its own named on-disk store (and clears its namespaced selection UserDefaults key); `relaunchPreservingStore()` relaunches without `--reset` to assert persistence. Query elements by `AccessibilityID` constants. Use the base class's drag helper rather than raw `press(forDuration:thenDragTo:)`.

## Pitfalls (hard-won — check before "fixing")

- **NSUndoManager in unit tests:** no run loop is pumped, so `groupsByEvent` never opens a group; a `registerUndo` outside explicit begin/end grouping throws, and under Swift Testing that **hangs the runner forever**. This rule is asymmetric with the running app: `TestContainer` sets `groupsByEvent = false` for headless unit tests (no run loop to open event groups), but the app's window undo manager keeps the default `groupsByEvent = true` — forcing it false there crashed on-disk SwiftData registrations that happen outside BoardStore's explicit groups (autosave, relationship maintenance; see the M7 crash evidence at `RootView.wireUndoManager`). `BoardStore.init`'s own `groupsByEvent = false` line is therefore a no-op in the app (the window's undo manager is attached later, in `RootView.wireUndoManager`) and only takes effect in tests. What both modes actually rely on is `BoardStore`'s explicit per-operation grouping; keep that when adding undoable mutations, and use `TestContainer(withUndo: true)` in tests.
- **Board delete is not undoable** — SwiftData fatally asserts when undo-snapshotting an on-disk cascade delete of a Board. `BoardStore.deleteBoard` detaches the undoManager around the delete and clears the stack; the PRD documents this. Don't "fix" it by re-enabling undo there. **M-E extended this to card and list deletes** (checklist-cascade spike, `ChecklistUndoOnDiskTests`): all three deletes use the detach-and-clear discipline. Don't "re-enable" undo on any of them. **M-F extended it again to area deletes** (`AreaUndoOnDiskTests`): all four deletes use detach-and-clear. `setArea` (board-to-area move) spiked RED too but for a *different* reason — SwiftData's automatic undo registration silently drops relationship membership, not a crash — so it keeps undoability via a manual-inverse `registerUndoable` pair instead of detaching; don't conflate "spiked RED" with "not undoable" for that one operation.
- **JSON import (E-02) is not undoable, in either mode** — the on-disk undo spike (`ImportUndoOnDiskTests`) was never reached: during Task 1's TDD cycle, the in-memory `BoardStoreImportTests.appendIsOneUndoStep` crashed deterministically (3/3 identical runs) on `redo()` — SwiftData's automatic undo registration restored the Board and BoardList levels of a multi-board imported graph but **silently dropped every third-level Card insert** (the crash was an index into the resulting empty `sortedCards` — missing rows, not reordering; the `moveCard` broken-redo precedent one level deeper). That's the spike's own failure criterion (silent wrong-state-after-redo, not just a crash), observed in the *weaker* in-memory environment, so the fallback was enacted without ever running the on-disk leg (`ImportUndoOnDiskTests` ships instead as an on-disk materialize smoke, not an undo/redo regression). Both `BoardStore.importBoards` (Add to Existing) and `replaceAllBoards` (Replace All) use the exact `deleteBoard` detach-and-clear discipline — undoManager detached for the whole materialize span, reattached and `removeAllActions()`'d in a `defer`. ⌘Z after any import does nothing; don't "fix" this by wrapping import in a `withUndoGroup`. Same spike also surfaced a test-oracle bug worth remembering elsewhere: `ObjectIdentifier` is **not** a stable SwiftData identity oracle across saves (instances refault, so PASS/FAIL varied run-to-run) — identity assertions on model rows must use `persistentModelID`. M-F's `setArea` (board-to-area move) is the second user of the manual-inverse `registerUndoable` pattern first shipped for `moveCard`'s cross-list reassignment: automatic undo registration disabled for the span of the mutation, an explicit inverse pair registered instead — undoability preserved even though the operation spiked RED for automatic registration.
- **No window on `--uitest` launch:** macOS doesn't auto-present the WindowGroup window under XCUITest; `TackUITestCase.ensureWindow` (⌘N nudge) handles it — always launch through the base class. The same applies to **manual** `open … --args --uitest` launches on this host: the instance lands windowless, and with no window `RootView` never exists, so every board-dependent menu item is disabled — presenting as "clicking the menu does nothing" (burned E-02's manual gate, 2026-07-08; with a window present both file panels present fine). Any manual procedure using the scratch-store launch must make File ▸ New Tack Window (⌘N) its second step.
- **Verifying panel/dialog presentation without XCUITest:** `osascript`/System Events can click Tack's real menu items and count `sheets of window 1` — enough to prove a save/open panel or mode dialog actually presents (this cleared E-02's "Export does nothing" report in minutes). Panel *contents* stay unreachable (remote-hosted) — this verifies presentation, never interaction.
- **`.accessibilityIdentifier` on an ancestor shadows child ids** — never identify an ancestor of a container you query by id (this is why `RootView` hangs `root-view` off a sibling `Color.clear`).
- **Drag retries must poll the postcondition** before retrying; an instant `.exists` check triggers a spurious second drag that corrupts state.
- **`Reordering.movedWithin` has two overloads with different destination conventions:** `(from:to:)` takes the target index in the *resulting* array (used by `moveList`/`moveCard`); `(fromOffsets:toOffset:)` takes SwiftUI's *pre-removal* insertion offset (used by `moveBoards`). `.onMove` handlers must use the offsets overload — the index one silently off-by-ones down-moves.
- **Native `List` `.onMove` row-reorder cannot be driven by synthetic input** (confirmed for the B-06 sidebar): under XCUITest's `press(forDuration:thenDragTo:)` the row lifts and live-previews the reorder, but the drop never commits; CGEvent drags (posted at the HID event tap, `.cghidEventTap`) fail harder — they never even initiate the reorder preview — while the *same* CGEvent technique commits the board-canvas `Transferable` drag fine. So the limitation is NSTableView's native row-drag session under synthetic input (internal mechanism uninstrumented). Human-verified working with a real mouse (2026-07-08; 30-second procedure in the B-06 spec's Testing section). `SidebarReorderUITests` keeps only the filter-gate test; the reorder logic is unit-covered (`Reordering`, `BoardStore.moveBoards`), not e2e-covered.
- **Text-input focus detection:** `firstResponder` is useless in SwiftUI here (always a private proxy). The working pattern is the `textInputFocused` `@FocusedValue` published by every TextField/TextEditor, plus an action-level re-check including `NSApp.keyWindow?.isSheet`. New text fields must publish it or menu shortcuts will fire while typing.
- **The sidebar filter field must stay `.textFieldStyle(.roundedBorder)` and must NOT get a `.focused()` binding.** The NSTextField-backed style keeps its launch key-view grab at the AppKit level only; switching to `.plain` (SwiftUI-native) — or binding `FocusState` — bridges launch focus into SwiftUI, publishes `textInputFocused = true`, and silently disables the whole keyboard command surface at launch. Also, `BoardView`'s Esc → `.onExitCommand` (filter-bar hide) only fires via the scene-wide fallback dispatch that exists when NO SwiftUI view has focus: every attempted "release launch focus" workaround (`defaultFocus(false)`, `makeFirstResponder(nil)`/`contentView`, focusing the List) broke it — `LabelFilterUITests.testClearRestores` is the regression gate. The known cost: a manual/production launch starts with the field focused (keyboard commands dead until first click) — an open follow-up needing an AppKit `initialFirstResponder` approach, not SwiftUI focus machinery.
- **XCUITest measures sheets in clipped AX-space, not frame-space:** the UITest window clamps a sheet's visible height — a hard `.frame(height: 560)` measures 520 through `XCUIElement.frame`, and a flexible frame opens ~10pt wider than `idealWidth`. Never pin exact sheet dimensions in UI tests; use band assertions (see `testOpenDetailViaDoubleClickBody`'s opening-size pin, M-0).
- **A plain SwiftUI `Button` is ONE atomic AX element under XCUITest** — inner `Text`/`Image` children are not queryable descendants (`.staticTexts` inside it is always empty) and the button's `label` is synthesized from its content. Tests that must distinguish button *renderings* need a different oracle (M-0 used chip geometry: capsule ~99pt vs circle 26pt). Corollary: removing a visible `Text` from a button silently strips its accessible name unless you add `.accessibilityLabel`.
- **Decorative SwiftUI overlays over text editors must be `.allowsHitTesting(false)`.** A `strokeBorder` overlay on a `TextEditor` intercepted the click that gives the backing NSTextView keyboard focus — XCUITest `typeText` failed with "Neither element nor any descendant has keyboard focus" and the element read `Disabled` (caught by `testEditDescriptionSavesAndPersists`). Same rule for hairlines over the inline TextFields (caret clicks). Relatedly, `.scrollContentBackground(.hidden)` on macOS `TextEditor` makes it report AX-Disabled under XCUITest and is unnecessary anyway (the editor is already transparent).
- **SwiftUI `Section(header:)` in a macOS sidebar `List` flattens the hosted header's accessibility under XCUITest** — the whole header row collapses to ONE StaticText carrying only its chevron's id (observed: `area-toggle-Office`); the container's own id (`area-Office`) is absent entirely and the chevron Button is demoted out of existence as an interactive element. M-F's `AreaHeaderView` ships as the section's first ROW (`.selectionDisabled(true)`, header-styled text) instead of a `Section(header:)` view — rows preserve their hosted AX subtree, headers don't. Evidence: `.build/mf-task3-sectionheader-evidence.log` + the M-F plan's Amendments entry (2026-07-09). Don't move a sidebar section header back into `header:` without re-running `AreaUITests`.
