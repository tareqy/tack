# Areas (M-F) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Boards gain opt-in, collapsible sidebar groups called **Areas** — a new top-level SwiftData entity (`Area`: name, position, isCollapsed) with a `.nullify` relationship to Board (the app's first explicit nullify and Board's first to-one), a sectioned sidebar that renders ungrouped boards first (headerless, byte-identical to today when zero areas exist) followed by one collapsible section per area, membership moves via a context-menu "Move to Area ▸" submenu (the proven `CardView` "Move to List" pattern), and export `formatVersion` 5 carrying a real top-level `areas[]` plus a per-board `area` name reference, merged by exact trimmed name on import. This is the LAST milestone (M-F); the PRD gains B-07 and its §4.6/§6/§9 amendments. Before ANY of that: the milestone opens with a **SPIKE**, because area delete (a `.nullify` release of N boards at once) and board-to-area moves (a relationship reassignment across TWO `Area.boards` inverse collections) are exactly the multi-object-relationship-write shapes at which SwiftData's undo machinery has already failed this codebase three times (the on-disk Board-delete fatal assert; the import spike's silent third-level redo drops; `moveCard`'s cross-list broken redo). The spike's evidence, not anyone's optimism, decides the undo posture of `deleteArea`, `setArea`, and `createArea(named:moving:)`.

**Architecture:** Task 0 adds the entity (`Tack/Models/Area.swift`, joining `TackSchemaV1.models` — the ChecklistItem/`efc2e4c` additive precedent: new table, lightweight migration, no schema version bump, no migration stage) plus optional `Board.area`, the three store methods in their TRIAL `withUndoGroup` forms, and an on-disk spike suite `TackTests/AreaUndoOnDiskTests.swift` (this is the promised THIRD user of the private `OnDiskStore` helper, so Task 0 also promotes it to `TackTests/Helpers/OnDiskTestStore.swift` per the existing files' own "promotion can wait for a third user" note). `Board.position` stays ONE global `0..<n` sequence — the sidebar groups rows at render time — so `SelectionRestore`, `NextBoardSelection`, ⌘1–⌘9's `sortedBoards`, and all three `@Query(sort: \Board.position)` sites are untouched; the entire index cost lands in one new pure function, `Reordering.movedWithinSubset`, which translates a section-local `.onMove` into a permutation of the global order (unit-tested like every other `Reordering` function — never in the view). Export v5 restructures the envelope with an optional-defaulted-nil `areas: [ExportArea]?` (the `ExportCard.checklist` keyNotFound trap, verbatim) and `ExportBoard.area: String?`; `materialize` gains a find-or-CREATE area pre-pass (the CardLabel merge dictionary shape with the opposite missing-row policy — labels are a closed palette, areas are open user data) fed an explicit `existingAreas` parameter so `replaceAllBoards` (which now also deletes every Area — no empty areas survive a restore) can pass `[]` instead of fetching pending-deleted rows mid-span. The `standard` fixture gains exactly ONE area — "Office", expanded, wrapping the EXISTING Work board — so the load-bearing roster (names, count, global positions, flat ⌘1/⌘2 order) is byte-identical and every one of the 74 existing UI tests sees its rows exactly where they were, while area UI tests get a real area to assert against. Cross-section drag is CUT from scope entirely (see Design Decisions (e)); within-section drag inherits B-06's manually-verified posture.

**Tech Stack:** SwiftUI (macOS 14), SwiftData (versioned schema), XCUITest, Swift Testing, xcodegen.

## Design Decisions (locked here — do not relitigate mid-implementation)

**(a) Area delete = `.nullify` (release boards to ungrouped), never cascade.** Every populated on-disk cascade delete tried in this repo fatally asserts under undo snapshotting (board delete: crash reports cited at `BoardStore.swift:135-143`, including a board with ZERO cards; card/list deletes: the M-E spike, 3/3 crashes at `SwiftData/ModelSnapshot.swift:46`) — an Area→Board cascade would sit ABOVE the already-crashing shape AND destroy user boards wholesale, and `NextBoardSelection` has no multi-delete form. Nullify keeps `deleteBoard` as the app's destruction ceiling: area delete is a small relationship edit that boards survive. BUT nullify's N-board `area` rewrite is itself the known redo-drop shape (`moveCard`'s two-collection hazard, `BoardStore.swift:334-346`), so Task 0 is an on-disk undo spike for `deleteArea` AND `setArea`/`createArea(named:moving:)`, with pre-agreed fallbacks written into Task 1b: leg A RED → `deleteArea` adopts `deleteBoard`'s detach-and-clear verbatim (the fourth non-undoable delete); leg B RED → `setArea` adopts `moveCard`'s manual-inverse `registerUndoable` pattern (undoability preserved); leg C RED → `createArea(named:moving:)` splits into an undoable insert plus the shipped `setArea` form.

**(b) `Board.position` stays ONE global 0..<n sequence; the sidebar groups at render time.** Per-area scoping breaks four consumers that assume globally-unique positions — `SelectionRestore.resolve`'s min-by-position (`SelectionRestore.swift:19`), `NextBoardSelection.resolve` (`SelectionNavigation.swift:184`), `RootView.sortedBoards`/⌘1–⌘9 (`RootView.swift:245, 285-290`), and the three `@Query(sort: \Board.position)` sites (`SidebarView.swift:6`, `RootView.swift:12`, `TackApp.swift:176`), which cannot express a composite (areaPosition, boardPosition) sort over Board alone. Global scoping's entire cost is one pure function (`Reordering.movedWithinSubset`) plus one rule: a board keeps its global position when it changes area (the row just renders under a different header). The `SidebarView:24-29` "index convention handled in one place, nowhere in the view" contract is preserved — the view passes section-local `.onMove` arguments straight through to `BoardStore.moveBoards(fromOffsets:toOffset:in:)`.

**(c) ⌘1–⌘9 keeps flat global position order regardless of grouping or collapse; selecting/restoring a board inside a collapsed area auto-expands that area.** Zero code change to `AppCommands`/`RootView.selectBoard`/`boardNames` — `testCmd1Cmd2SwitchBoards` keeps passing verbatim (and, with Work now inside Office, exercises "⌘2 selects a board inside an area" for free). Auto-expand is ONE view-side effect in `RootView`'s existing `.onChange(of: selectedBoardID)` — routed through `store.setAreaCollapsed(area, false)`, an honest "Expand Area" undo step (the no-op guard makes every other selection change free) — and covers restore-at-launch, ⌘1–⌘9, post-delete `NextBoardSelection`, post-import selection, and create, from one site. Documented in the PRD: the detail pane always shows the selected board; auto-expand keeps the sidebar highlight visible too.

**(d) Area collapse state is MODEL state (`Area.isCollapsed`), the `BoardList.isCollapsed` precedent wholesale:** ships in the entity from day one, mutated only by `BoardStore.setAreaCollapsed` as one undo step ("Collapse Area"/"Expand Area"), included in the v5 export (`ExportArea.isCollapsed`), and — on merge into an EXISTING area — the local flag wins (importing a backup must not fold/unfold the sidebar you are looking at). The `BoardViewMode` UserDefaults alternative is rejected in one sentence: it would fork the collapse-persistence pattern the app already has (list collapse is model state that round-trips export), for the price of a new codec, a new namespaced defaults key, `--reset` plumbing, and a collapse state that silently dies with the machine instead of traveling with the backup.

**(e) Cross-area/board-to-area moves ship as a context menu — "Move to Area ▸" listing areas + "No Area" + "New Area…" — and cross-section DRAG is cut entirely.** The submenu-under-XCUITest pattern is proven in this exact repo: `CardView`'s "Move to List" submenu is driven end-to-end by `CardCRUDUITests.testMoveViaContextMenu` (`contextMenuItem("Move to List").click()` then `contextMenuItem("Done").click()`), so the primary UX is fully unit- AND UI-testable. Sectioned-`List` cross-section `.onMove` drag is uncharted, inherits the B-06 synthetic-input untestability, and SwiftUI's per-`ForEach` `.onMove` is section-local by construction — a spike could only confirm what the API shape already says. It is cut, listed as a possible follow-up in the PRD, and the human checklist verifies an attempted cross-section drag snaps back without corrupting order. Within-section drag reorder IS kept (per-section `.onMove` → `moveBoards(in:)`), unit-covered and manually verified like B-06.

**(f) Area creation/rename is a small native sheet (`AreaNameSheet`, the CreateBoardSheet/EditBoardSheet pattern), reached only through context menus; the Edit Board sheet is NOT touched.** Areas are created exclusively via "Move to Area ▸ New Area…" (an area always starts with a board; empty areas arise only by later releasing/deleting members and are deleted explicitly via their header menu) — a standalone "New Area" toolbar/menu item would create empty groups with no purpose and land in the sidebar-toolbar overflow trap (`RootView.swift:116-124`). `createArea(named:moving:)` is **find-or-create** by exact trimmed name, so typing an existing name in the New Area sheet simply moves the board there and duplicate area names are impossible via any path (duplicates would make import's merge-by-name ambiguous and merge silently on round-trip). Rename uses the same sheet (Save disabled on a name collision — rename-to-merge is out of scope). Adding an area picker to `EditBoardSheet` is rejected: it duplicates the context-menu affordance, grows `editBoard`'s signature and every call site, and bloats a sheet whose scope is board identity, not sidebar organization.

**Supporting decisions (each pinned by a test named below):** merge key is the **exact, case-sensitive, whitespace-trimmed name** — the CardLabel materialize precedent (`BoardStore.swift:601-647`) is the only in-repo *merge/attach-by-name* precedent and is exact-match; `filterBoards`' `localizedCaseInsensitiveContains` is a *search* affordance, and a merge whose behavior varies by system locale would be a data-integrity hazard. Ungrouped boards render FIRST, headerless — zero areas renders byte-identically to the pre-M-F sidebar. While the sidebar filter is non-empty the list renders FLAT (headerless, all boards including collapsed areas' members, no `.onMove` anywhere) — search must find what collapse hides, and this keeps the "reorder enabled ⇔ showing all boards" invariant structural. `replaceAllBoards` deletes every Area too (restore-the-backup-exactly; no stranded empty areas). New boards (⇧⌘N, sheet, empty state) are always ungrouped — `createBoard` is untouched. Areas have a `position` (creation order) but NO reorder UI in v1 (the M-E "no reorder UI" posture) and no `createdAt` (YAGNI — nothing reads it, export doesn't carry it). `EmptyStateView`'s `boards.isEmpty` gate is unchanged: with zero boards but surviving empty areas, the empty state shows beside the headers and its "Create your first board" copy stays truthful — accepted, documented in Task 3.

```swift
import Foundation
import SwiftData

/// M-F: an opt-in sidebar group of boards ("Area"). Single level, no nesting, no completion
/// semantics (PRD B-07/D-03). `position` is the area's order among areas (creation order —
/// no reorder UI in v1); `isCollapsed` is model state exactly like `BoardList.isCollapsed`
/// (undoable via `BoardStore.setAreaCollapsed`, exported in formatVersion 5). The `boards`
/// relationship is the app's FIRST explicit `.nullify`: deleting an Area releases its boards
/// to ungrouped — it never deletes boards (see `BoardStore.deleteArea`). Additive entity in
/// TackSchemaV1 — a new table is a lightweight migration, the ChecklistItem posture; an M-F
/// store opened by an older build simply ignores the table.
@Model
final class Area {
    @Attribute(.unique) var id: UUID
    var name: String
    var position: Int
    var isCollapsed: Bool

    @Relationship(deleteRule: .nullify, inverse: \Board.area)
    var boards: [Board]

    init(
        id: UUID = UUID(),
        name: String,
        position: Int,
        isCollapsed: Bool = false,
        boards: [Board] = []
    ) {
        self.id = id
        self.name = name
        self.position = position
        self.isCollapsed = isCollapsed
        self.boards = boards
    }

    var sortedBoards: [Board] { boards.sorted { $0.position < $1.position } }
}
```

```swift
    /// M-F: find-or-CREATE an area by its EXACT whitespace-trimmed name (case-sensitive — the
    /// ONE merge key shared with import's area merge and AreaNameSheet's rename gate; exact
    /// match is the CardLabel attach-by-name precedent, and a locale-sensitive comparison would
    /// make merges machine-dependent). A matching existing area is reused — so duplicate area
    /// names are impossible via any path, and "New Area…" with an existing name simply moves
    /// the board there. On create, `board` (when non-nil) joins the area in the SAME "New Area"
    /// undo step, so the context menu's New Area… gesture costs one ⌘Z. Returns nil only for a
    /// whitespace-only name (the sheet disables Save; this is the store backstop).
    /// TRIAL FORM (M-F Task 0 spike, leg C): the insert+relationship write runs under one
    /// withUndoGroup; the spike verdict decides whether it stays (Task 1a) or splits (Task 1b).
    @discardableResult
    func createArea(named name: String, moving board: Board?) -> Area? {
        let trimmed = name.trimmingCharacters(in: .whitespacesAndNewlines)
        guard !trimmed.isEmpty else { return nil }
        if let existing = fetchAreas().first(where: { $0.name == trimmed }) {
            if let board { setArea(board, to: existing) }
            return existing
        }
        var created: Area!
        withUndoGroup("New Area") {
            let position = (fetchAreas().map(\.position).max() ?? -1) + 1
            let area = Area(name: trimmed, position: position)
            context.insert(area)
            if let board { board.area = area }
            save()
            created = area
        }
        return created
    }

    /// M-F: moves a board between areas (nil = ungrouped). Rewrites ONLY `board.area` — the
    /// global `Board.position` is untouched (design (b)): the row renders under a different
    /// header, the total order of boards is unchanged, so ⌘1–⌘9 / SelectionRestore /
    /// NextBoardSelection never notice. One undo step; a same-area call is a whole-call no-op
    /// (opens no group, does not save).
    /// TRIAL FORM (M-F Task 0 spike, leg B): a relationship reassignment across up to TWO
    /// Area.boards inverse collections — the moveCard cross-list hazard shape one level up.
    func setArea(_ board: Board, to area: Area?) {
        guard board.area?.id != area?.id else { return }
        withUndoGroup(area == nil ? "Remove from Area" : "Move to Area") {
            board.area = area
            save()
        }
    }

    /// M-F: deletes the area; the `.nullify` rule releases its boards to ungrouped — boards are
    /// NEVER deleted and their global positions are untouched, so area delete's blast radius is
    /// strictly smaller than a single board delete.
    /// TRIAL FORM (M-F Task 0 spike, leg A): an N-board relationship release under undo
    /// snapshotting. Task 1a keeps this withUndoGroup form (verdict GREEN) or Task 1b rewrites
    /// it to deleteBoard's detach-and-clear (verdict RED — the pre-agreed fallback).
    func deleteArea(_ area: Area) {
        withUndoGroup("Delete Area") {
            context.delete(area)
            save()
        }
    }
```

```swift
/// → deleteBoard's detach-and-clear, non-undoable, stack-wiping; leg B (setArea) → moveCard's
/// manual-inverse registerUndoable pattern; leg C (createArea moving a board) → the area insert
/// keeps withUndoGroup and the board move routes through setArea's shipped form afterwards.

    /// Boards Alpha/Beta/Gamma (global positions 0/1/2); Alpha and Beta grouped into "Home";
    /// Gamma stays ungrouped as the untouched control. Stack CLEARED after seeding so the only
    /// group under test is the operation itself.
```

`Tack/Store/Reordering.swift` — after the `movedWithin(_:fromOffsets:toOffset:)` overload:

```swift
    /// M-F: reorders the members of `subset` among the SLOTS they currently occupy in `items`,
    /// leaving every non-member at its exact index — the sectioned-sidebar reorder primitive
    /// (Board.position stays ONE global sequence; a per-area `.onMove` hands offsets into ITS
    /// section's rows, i.e. into `items.filter { subset.contains($0) }`, using the same SwiftUI
    /// pre-removal convention as `movedWithin(fromOffsets:toOffset:)`). Total: out-of-range
    /// offsets follow that overload's rules, a subset covering all of `items` IS that overload
    /// (pinned by ReorderingTests), and foreign subset members are simply never encountered.
    static func movedWithinSubset<Element: Hashable>(
        _ items: [Element], subset: Set<Element>,
        fromOffsets source: IndexSet, toOffset destination: Int
    ) -> [Element] {
        let members = items.filter { subset.contains($0) }
        let reordered = movedWithin(members, fromOffsets: source, toOffset: destination)
        var iterator = reordered.makeIterator()
        return items.map { subset.contains($0) ? (iterator.next() ?? $0) : $0 }
    }
```

```swift
    /// section. `source`/`destination` are SwiftUI `.onMove` offsets INTO THAT SECTION's rows
    /// (the position-sorted members), passed straight through by `SidebarView` — the index

    /// by `deleteBoard`, as before); non-members keep their exact slots. Identity moves return
    /// BEFORE opening an undo group, so "drop it back where it was" never eats a ⌘Z step.
    /// `in:` is deliberately NOT defaulted — a defaulted nil would silently reinterpret a
    /// flat-offset call as "the ungrouped section"; every call site states its section.
    func moveBoards(fromOffsets source: IndexSet, toOffset destination: Int, in area: Area?) {

                                                    fromOffsets: source, toOffset: destination)
        guard newOrder != ids else { return }
        withUndoGroup("Move Board") {
            applyPositions(newOrder, to: boards)
            save()
        }
```

```swift
/// M-F: the one small sheet for area names (the CreateBoardSheet/EditBoardSheet pattern) —
/// create mode is reached only via "Move to Area ▸ New Area…" (an area always starts with a
/// board; see Design Decisions (f)), rename via the area header's context menu.
struct AreaNameSheet: View {
    enum Mode {
        /// Creates (or finds — createArea is find-or-create) an area and moves `board` into it.
        case create(moving: Board)
        case rename(Area)
    }

    let mode: Mode
    let store: BoardStore

    @Environment(\.dismiss) private var dismiss
    @Query(sort: \Area.position) private var areas: [Area]
    @State private var name: String

    init(mode: Mode, store: BoardStore) {
        self.mode = mode
        self.store = store
        switch mode {
        case .create: _name = State(initialValue: "")
        case .rename(let area): _name = State(initialValue: area.name)
        }

        name.trimmingCharacters(in: .whitespacesAndNewlines)

    /// deliberately allows collisions — createArea's find-or-create turns "New Area… 'Office'"
    /// into a plain move, which is the honest reading of the gesture. The collision key is THE
    /// merge key: exact, trimmed, case-sensitive.
    private var renameCollides: Bool {
        guard case .rename(let area) = mode else { return false }
        return areas.contains { $0.name == trimmedName && $0.id != area.id }
    }

    private var title: String {
        switch mode {
        case .create: "New Area"
        case .rename: "Rename Area"
        }

        VStack(alignment: .leading, spacing: 16) {
```
